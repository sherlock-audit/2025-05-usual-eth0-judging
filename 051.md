# Uninitialized mintCap leads to all `swap()` calls reverting unless `setMintCap()` is explicitly called

### Summary

Uninitialized `mintCap` leads to all `swap()` calls reverting unless `setMintCap()` is explicitly called

### Root Cause

In [`Eth0.sol`](https://github.com/sherlock-audit/2025-05-usual-eth0/blob/main/eth0-protocol/src/token/Eth0.sol), `mintCap` is not set anywhere, neither in the `initialize()` function nor in the constructor. This means it will be 0 for an undefined amount of time.

`DaoCollaterl::swap()` calls `Eth::mint()`, which has [this check](https://github.com/sherlock-audit/2025-05-usual-eth0/blob/main/eth0-protocol/src/token/Eth0.sol#L140):
```solidity
        // Check if minting would exceed the mint cap
        if (totalSupply() + amount > $.mintCap) {
            revert AmountExceedCap();
        }
```

`mintCap` will be 0, so `swap()` will always revert since that condition will always be true.


### Internal Pre-conditions

`setMintCap` has not been called.

### External Pre-conditions

N/A

### Attack Path

Not an attack, but the issue can happen as follows:
- `setMintCap` has not been explicitly called.
- Users try to call swap, but it'll always revert.

### Impact

Prevents users from using the protocol temporarily, since `swap()` is a critical function.

    function _signPermitData(address token, SigUtils.Permit memory permit, uint256 ownerPrivateKey)
        internal
        returns (uint8, bytes32, bytes32)
    {
        IERC20Permit ercPermit = IERC20Permit(token);
        SigUtils sigUtils = new SigUtils(ercPermit.DOMAIN_SEPARATOR());
        bytes32 digest = sigUtils.getTypedDataHash(permit);
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(ownerPrivateKey, digest);
        return (v, r, s);

Initialized mintCap in the `initialize()` function.
